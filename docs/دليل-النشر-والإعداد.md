# دليل النشر والإعداد — نظام مجموعة الذيابي للمشتريات

مرجع موحّد لتشغيل النظام بالكامل: قاعدة البيانات، أسرار البيئة، البريد، والتحقق.
نفّذ الأقسام بالترتيب. (الأقسام المُعلّمة ✅ كافية للتشغيل الأساسي.)

---

## 1) أسرار البيئة في Cloudflare Pages ✅
المسار: **Cloudflare → Pages → المشروع → Settings → Environment variables** (اجعلها **Secret**).

| المتغيّر | مطلوب لِـ | مثال / ملاحظة |
|---|---|---|
| `GEMINI_API_KEY` | القراءة الذكية للوثائق + مساعد الذكاء | مفتاح Google AI Studio |
| `SUPABASE_URL` | إدارة المستخدمين + إشعارات البريد | `https://xxxx.supabase.co` |
| `SUPABASE_SERVICE_ROLE_KEY` | إدارة المستخدمين + إشعارات البريد | مفتاح service_role (**سرّ — لا يُكشف للعميل**) |
| `RESEND_API_KEY` | إشعارات البريد | مفتاح resend.com |
| `NOTIFY_FROM` | إشعارات البريد | `موردو الذيابي <noreply@aldeyabi.com>` |
| `NOTIFY_REPLY_TO` | (اختياري) | `supply@aldeyabi.com` |
| `WATHQ_API_KEY` | (اختياري) التحقق من السجل التجاري | مفتاح واثق |

> بعد أي إضافة/تعديل لمتغيّر، نفّذ **Retry deployment** ليُحمّل.

إن لم تُضبط أسرار البريد، يتجاهل النظام الإرسال بهدوء دون تعطيل التسجيل أو المراجعة.

---

## 2) قاعدة البيانات (Supabase → SQL Editor)
شغّل الملفات **بهذا الترتيب** (كلها قابلة لإعادة التشغيل بأمان):

| # | الملف | الغرض |
|---|---|---|
| 1 | `db/workflows.sql`, `db/rfq.sql`, `db/rfq-portal-v2.sql`, `db/purchase-orders.sql` | إنشاء الجداول/السياسات الأساسية (إن لم تكن منشأة) |
| 2 | `db/secure-resume.sql` | دوال استئناف الطلب الآمنة (RPC) |
| 3 | `db/registration-status.sql` | **دالة متابعة المورد لحالته** (لتعمل نافذة «تتبّع طلبي») |
| 4 | `db/hardened-rls.sql` ✅ | **تحصين سياسات RLS** (المرحلة 0 + المرحلة 1) |

> **متطلّب المرحلة 1 من التحصين:** يجب أن يسجّل موظفو لوحة الإدارة الدخول عبر **Supabase Auth**
> (مُفعّل في التطبيق عبر `signInWithPassword`) قبل التطبيق، وإلا تتوقّف القراءة/الكتابة في اللوحة.

### حاوية التخزين
أنشئ حاوية **`supplier-docs`** خاصة (public = false) من Dashboard → Storage، وطبّق سياسات
`storage.objects` الموضّحة في نهاية `db/hardened-rls.sql` أو `db/security-hardening.sql`.

---

## 3) إعداد البريد (Resend) ✅
1. أنشئ حساباً مجانياً في **resend.com** وأصدر **API Key**.
2. وثّق نطاق `aldeyabi.com`: Resend → **Domains → Add Domain**، ثم أضِف سجلّي **SPF + DKIM** في DNS.
3. اضبط المتغيّرات `RESEND_API_KEY` و`NOTIFY_FROM` (انظر القسم 1).
4. أعد النشر.

**القوالب** (ثنائية اللغة عربي/إنجليزي) تُرسَل تلقائياً عند:
الاستلام · القبول · الرفض · يحتاج تعديل — من `functions/api/notify.js`.

> الباقة المجانية (100/يوم، 3000/شهر) تكفي حجم إشعارات التسجيل بسهولة.

---

## 4) التحقق من التشغيل
- افتح `https://<موقعك>/api/ai` → يجب أن يردّ JSON (الذكاء مهيّأ).
- افتح `https://<موقعك>/api/notify` → يجب `{"ok":true}` (البريد مهيّأ).
- من بوابة التسجيل: أرسل طلباً تجريبياً → يصلك بريد «الاستلام»، وجرّب **«تتبّع طلبي»** (رقم الطلب + سجل تجاري/بريد).
- من نظام المشتريات: غيّر حالة الطلب (قبول/إرجاع) → تصل رسالة الحالة المناسبة، وتظهر للمورد في المتابعة.
- تأكّد من: تبديل اللغة عربي/إنجليزي في البوابة، قراءة OCR لمستند، وظهور الشعار كاملاً.

---

## 5) ملاحظات أمنية
- مفاتيح `service_role` و`RESEND` و`GEMINI` تبقى على الخادم فقط (أسرار Cloudflare) — لا تُكشف للمتصفح إطلاقاً.
- جدول `proc_supplier_registrations`: **إدراج للزوّار فقط**؛ القراءة/التعديل للموظفين المسجّلين فقط.
  متابعة/استئناف المورد تتمّان عبر دوال `SECURITY DEFINER` محقّقة، لا عبر فتح الجدول.
- نقطة `/api/notify` ليست مُرسِلاً مفتوحاً: same-origin + المستقبل من بريد الطلب فقط + قوالب ثابتة +
  تطابق الحالة + اشتراط جلسة موظف لإشعارات القرار.
- بعد ضبط `GEMINI_API_KEY` كسرّ، تأكّد من إزالة أي مفتاح قديم من `proc_settings` (المرحلة 0).
