# حزمة اختبار البوابة (النظام 3)

اختبارات انحدار دائمة **داخل المستودع** (بدل بيئة scratchpad المؤقتة) — تُشغَّل آلياً
على كل PR/دفع يمسّ البوابة عبر `.github/workflows/portal-tests.yml`.

## المحتوى
| الملف | الغرض |
|-------|-------|
| `00_roles.sql` | أدوار Supabase المحاكاة (anon/authenticated/service_role/authenticator) على مستوى العنقود. |
| `10_outbox.sql` | صندوق الصادر المعامَلاتي (029): التقاط · idempotency · claim · منع سحب مزدوج · تراجع أُسّي · dead-letter · sent · purge (8 تأكيدات). |
| `11_security.sql` | التصليب الأمني (030) + نموذج الصلاحيات: anon محجوب · authenticated سليم · search_path · عدم كسر الدورة (7 تأكيدات). |
| `12`–`19` | تأكيدات المحارس على مستوى الوحدة (ميزانية/آيبان/مطابقة ثلاثية/مرتجعات/عملات/عقود/ترسية مجزّأة/تكامل الفاتورة). |
| `20_scenarios_lifecycle.sql` | **اختبار تكامل على مستوى سير العمل** عبر الـRPC الحقيقية بهوية مُنتحَلة (JWT): تأجيل مالي/استئناف · تفويض عند الغياب · ترسية مجزّأة + صرف لكل مورد · استلام جزئي متعدّد · شريحة اللجنة في أمر الشراء · إرجاع المشتريات للمقدّم (23 تأكيداً). |
| `21_scenarios_advanced.sql` | سيناريوهات متقدّمة عبر الـRPC: سلسلة أمر شراء ثلاثية (لجنة→مالية→مدير عام) بفصل مهام كامل · الدفعات على مراحل · مرتجع + إشعار مدين (12 تأكيداً). |
| `22_jobs_coverage.sql` | تغطية الأدوار (042): كل صلاحية حرِجة تمنحها وظيفة نشطة ≥1 · gm/qc غير فارغتين · can_disburse بوظيفتين لفصل مهام الصرف (5 تأكيدات). |
| `23_rework_edit.sql` | الإعادة الذكية للتصحيح (043): إرجاع→تعديل فعلي (بنود/إجمالي/revision)→إعادة تقديم→إعادة اعتماد كاملة + حُرّاس (المقدّم/can_edit فقط · returned فقط) (7 تأكيدات). |
| `run.sh` | يحمّل `portal-standalone.sql` كاملاً في قاعدة نظيفة ثم يشغّل التأكيدات (99 إجمالاً). أي فشل ⇒ خروج غير صفري. |

> **ملاحظة عن ملفّي 20/21:** يختبران المنطق عبر **التسلسل الفعلي للدوال** (create→approve→award→
> po→payment→receipt) لا بالحقن المباشر، بانتحال الهوية عبر ضبط `request.jwt.claims.email`
> (تطابقه `portal_username()`) وكعب `auth.jwt()` المطابق لدلالات Supabase. هذا يثبت أنّ **سير العمل
> نفسه** يعمل، لا مجرّد المحارس المنفردة.

كل الاختبارات صيغة **تأكيدات** (`RAISE EXCEPTION` عند الخطأ) مع `ON_ERROR_STOP=1`،
فتُفشِل البناء تلقائياً.

## التشغيل محلياً
```bash
# على عنقود PostgreSQL قائم (مثال: socket /tmp/pt منفذ 5455)
PGHOST=localhost PGPORT=5455 PGUSER=postgres bash db/portal-tests/run.sh
```

## التوسّع
عند إضافة هجرة جديدة بمنطق حرِج: أضِف ملف تأكيدات `NN_*.sql` هنا وأدرجه في `run.sh`،
واحرص أن يُدمج تغيير المخطّط في `portal-standalone.sql` (التنصيب النظيف الذي تحمّله الحزمة).
