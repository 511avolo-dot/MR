# نظام المشتريات — مجموعة الذيابي

تطبيق ويب ثابت يتكوّن من نظامين:

| الملف | النظام | الوصول |
|------|--------|--------|
| `index.html` | النظام الداخلي الموحّد: متابعة الأسعار · الموردون · RFQ · الاعتمادات · **أوامر الشراء** | داخلي (تسجيل دخول) |
| `register.html` | تسجيل الموردين | عام |
| `rfq.html` | تقديم الموردين لعروض الأسعار (RFQ) | عام (برمز) |

> **وحدة أوامر الشراء (جديدة):** متابعة ذكية كاملة لأوامر الشراء (سجل + لوحة Kanban + تحليلات + إنذارات + استيراد/تصدير Excel)، بديل رقمي كامل لملف الإكسل.
>
> **تشغيل سكربتات Supabase (مرة واحدة، بالترتيب):**
> 1. `db/purchase-orders.sql` — إنشاء جدول `proc_purchase_orders` + RLS + الفهارس.
> 2. `db/purchase-orders-seed.sql` — ترحيل 53 أمر شراء حقيقيًا من الإكسل.
> 3. `db/suppliers-seed.sql` — ترحيل 42 موردًا (دمج بالاسم) إلى `proc_suppliers`.
>
> البيانات الحقيقية مزروعة أيضًا داخل التطبيق (بذور) فتظهر فورًا دون اتصال؛ والسكربتات تنقلها إلى السحابة.

**البنية:** HTML/CSS/JS مضمّن · قاعدة بيانات Supabase · ذكاء اصطناعي Google Gemini عبر وسيط خادم · نشر Cloudflare Pages.

## 🔐 معمارية الأمان (بعد التحصين)

### مفتاح Gemini API — على الخادم فقط
لم يعد المفتاح يُخزّن في قاعدة البيانات ولا يصل المتصفح. تمرّ كل طلبات الذكاء
الاصطناعي عبر وسيط Cloudflare Pages Function في `functions/api/ai.js`، الذي
يقرأ المفتاح من سرّ البيئة `GEMINI_API_KEY`.

**الإعداد (مرة واحدة):**
1. Cloudflare Dashboard → Pages → المشروع → **Settings → Environment variables**.
2. أضف متغيّراً: الاسم `GEMINI_API_KEY`، والقيمة مفتاح Google AI Studio. اجعله **Secret**.
3. أعد النشر.
4. شغّل `db/hardened-rls.sql` (المرحلة 0) لإزالة أي مفتاح قديم من جدول `proc_settings`.

> العميل يكتشف الوسيط تلقائياً عبر `GET /api/ai`. عند وجود الوسيط تُستخدم نقطة
> النهاية الآمنة؛ وإن أدخل مسؤول مفتاحاً محلياً (وضع تطوير) يُستدعى Gemini مباشرة
> عبر ترويسة `x-goog-api-key` (وليس في رابط الطلب).

### حدود الوسيط
- يقبل نماذج من قائمة بيضاء فقط، وإجراءَي `generateContent` / `streamGenerateContent`.
- يرفض الطلبات من مصادر مختلفة (فحص Origin) وحجم الطلب فوق 16MB.
- **متابعة موصى بها:** أضِف Cloudflare Turnstile أو مصادقة لمنع إساءة استهلاك الحصّة.

### سياسات قاعدة البيانات (RLS)
الوضع الافتراضي للنظام كان `USING (true)` على كل الجداول — أي وصول كامل لأي
حامل لمفتاح anon. ملف `db/hardened-rls.sql` يصحّح ذلك:
- **المرحلة 0 (فورية، غير كاسرة):** إزالة المفتاح السري من `proc_settings`.
- **المرحلة 1 (تحصين كامل):** قَصْر القراءة/الكتابة على مستخدمي Supabase Auth،
  مع إبقاء **إرسال** طلبات التسجيل متاحاً للعموم فقط (حماية بيانات الموردين).

> ⚠️ المرحلة 1 تتطلب أولاً ترحيل تسجيل دخول لوحة الإدارة إلى **Supabase Auth**
> (انظر أدناه)، لأن اللوحة تعمل حالياً بمفتاح anon.

## 🪪 ترحيل المصادقة (الخطوة المتبقية لإغلاق RLS)

النظام يصادق حالياً على مستوى الواجهة فقط (هاش SHA-256 في جدول `proc_users`
يُقرأ بمفتاح anon). هذا لا يحمي البيانات لأن مفتاح anon يمنح وصولاً مباشراً
للجداول بغضّ النظر عن شاشة الدخول.

**تم تنفيذ كود الترحيل بالفعل** في `index.html`: تسجيل الدخول يستخدم الآن
`supabase.auth.signInWithPassword`، وإدارة الجلسة والخروج عبر Supabase Auth،
والملف التعريفي (الدور/الصلاحيات) يُقرأ من `proc_users` بعد المصادقة.

يبقى عليك تنفيذ خطوات لوحة Supabase (≈6 دقائق) — راجع الدليل الكامل:
**[`db/auth-migration.md`](db/auth-migration.md)**

باختصار:
1. تفعيل موفّر «Email» وإيقاف تأكيد البريد.
2. إنشاء حسابات الدخول (بريد اصطلاحي `username@aldeyabi.local`).
3. بذر ملفات التعريف في `proc_users` (SQL في الدليل).
4. اختبار الدخول، ثم تطبيق المرحلة 1 من `db/hardened-rls.sql`.

> ⚠️ نفّذها على **مشروع Supabase تجريبي أولًا** للتأكد من الدخول قبل إغلاق RLS،
> تفاديًا لقفل نفسك خارج النظام. النطاق الاصطلاحي قابل للتعديل عبر الثابت
> `AUTH_EMAIL_DOMAIN` في `index.html`.

## 🚀 النشر

النشر على **Cloudflare Pages** (يدعم الوسيط `/functions` وترويسات `_headers`):

```
.github/workflows/deploy.yml   → Cloudflare Pages (push على main)
```

الأسرار المطلوبة في مستودع GitHub: `CF_API_TOKEN`، `CF_ACCOUNT_ID`.
وفي مشروع Cloudflare Pages: `GEMINI_API_KEY`.

> **توصية:** يوجد أيضاً `pages.yml` للنشر على GitHub Pages. لكن GitHub Pages
> **لا يشغّل** وسيط `/api/ai` ولا يطبّق ترويسات `_headers`؛ لذا يُنصح باعتماد
> Cloudflare Pages مصدراً وحيداً، أو تعطيل سير عمل GitHub Pages لتفادي نشر
> نسخة بميزات أمان منقوصة.

## 👥 إدارة المستخدمين (اختياري — للوحة إدارة كاملة)

دالة `functions/api/admin-users.js` تتيح إنشاء/حذف المستخدمين وإعادة تعيين
كلمات المرور وتعطيلها من داخل الواجهة، دون وضع مفتاح `service_role` في المتصفح.

**الإعداد:** أضف سرّين في Cloudflare Pages → Environment variables:
- `SUPABASE_URL` — رابط المشروع.
- `SUPABASE_SERVICE_ROLE_KEY` — مفتاح service_role (**سرّ** — لا يُكشف أبداً).

الأمان: كل طلب يتحقّق من رمز جلسة المستدعي ويتأكد أنه **مدير نشط** قبل التنفيذ.
إن لم تُنشَر الدالة، يتراجع التطبيق تلقائياً (تُدار كلمات المرور من لوحة Supabase).

## 🔁 استئناف طلب التسجيل (بعد إغلاق RLS)

بعد المرحلة 1 من RLS يُمنع anon من قراءة جدول التسجيل. دالتا RPC في
`db/secure-resume.sql` تتيحان للمورد استئناف طلبه عبر (id + token) السرّي فقط،
دون فتح الجدول. **شغّل هذا السكريبت قبل تطبيق المرحلة 1** إن كنت تستخدم الاستئناف.

## 🔄 سير العمل: اعتمادات الشراء + الإشعارات (داخل النظام)

مبنية بالكامل داخل Supabase — بلا مورّد خارجي. شغّل `db/workflows.sql` لإنشاء
الجدولين (`proc_purchase_requests`, `proc_notifications`) وحدّ الاعتماد.

- **طلب اعتماد**: أي مستخدم يرسل طلب شراء (وصف + مورد + مبلغ) من زر الجرس → «الاعتمادات».
- **التوجيه التلقائي**: حتى حدّ المبلغ (افتراضي 50,000 ر.س، قابل للتعديل في
  `proc_settings.approval_threshold`) → **مدير المشتريات**؛ وأكثر → **المدير العام**.
- **الإشعارات**: جرس داخل الواجهة يصل المعتمِد عند طلب جديد، ويصل مُقدّم الطلب عند القرار.
- **تعيين المعتمدين**: من إدارة المستخدمين، امنح `can_approve_l1` (مدير المشتريات)
  و`can_approve_l2` (المدير العام). الأخير يعتمد كل المستويات.

> ملاحظة: التحكّم على مستوى التطبيق (متّسق مع بقية النظام)؛ يمكن تحصينه لاحقاً
> بدوال RPC على الخادم. تفاصيل سياسة الموافقات قابلة للتوسعة لاحقاً.

## 📨 طلبات عروض الأسعار (RFQ) — بمستوى المؤسسات

شغّل `db/rfq.sql`. من زر الجرس → «عروض الأسعار». مبني على نهج Oracle Sourcing/Ariba:

- **تكامل مع الكتالوج**: اختر البنود من أصنافك ليُجلب **السعر المرجعي** تلقائياً من
  تاريخ النظام (last/avg price) كميزانية أساس للمقارنة.
- **اقتراح موردين ذكي**: من السجل التاريخي حسب أكواد/فئات البنود.
- **مصفوفة مقارنة**: تُبرز الأدنى لكل بند، وتُلوّن الانحراف عن السعر المرجعي
  (أخضر أقل / أحمر أعلى)، وتحسب **الوفورات %** لكل مورد مقابل المرجعي.
- **تقييم القيمة الأفضل (best-value)**: أوزان قابلة للضبط (سعر/تسليم/جودة/سداد)
  → درجة موزونة وترتيب، لا الأقل سعراً فقط.
- **توصية الترسية المثلى**: مقارنة بين **مورد واحد** و**ترسية مجزّأة** (كل بند
  لأوفر مورد) مع حساب التوفير الإضافي.
- **دورة حياة**: مفتوح/مغلق/مُرسى/ملغى + متابعة استجابة الموردين والمواعيد.
- **ترابط تكاملي**: الترسية (مفردة أو مجزّأة) تُنشئ **طلب/طلبات اعتماد** تلقائياً.
- الصلاحية: `can_manage_rfq`.

### بوابة المورّد (تقديم ذاتي بلا حساب)
شغّل `db/rfq-portal.sql`. صفحة عامة `rfq.html` (جوال أولاً، عربية، بسيطة):
- من شاشة RFQ → «روابط دعوة الموردين» → **انسخ رابط كل مورد** وأرسله عبر واتساب/إيميل.
- يفتح المورّد الرابط (برمز سرّي، بلا تسجيل دخول) ويُدخل أسعاره لكل بند، مع
  خيار «لا أعرض هذا البند»، وشروط (تسليم/سداد/صلاحية/جودة) وملاحظات.
- يُرسل العرض أو يحفظه مسودة، ويعدّله قبل الموعد. **تظهر أسعاره في المقارنة تلقائياً**.
- متابعة الحالة لكل مورد: مدعو / فتح الرابط / قدّم العرض.
- **مراعاة الموردين الصغار** (بطاقات بسيطة على الجوال، عربية، بلا حساب) **والكبار**
  (شروط منظَّمة، تعديل حتى الموعد). الوصول محمي بـ RPC مُتحقَّق بالرمز (SECURITY DEFINER).

## 🧩 وحدات المشتريات الذكية (داخل النظام)

| الوحدة | الوصف | الملف/التفعيل |
|--------|-------|----------------|
| اعتمادات الشراء | مستويان (مشتريات/عام) + توجيه تلقائي + إشعارات | `db/workflows.sql` |
| RFQ (مؤسسي) | كتالوج/خط أساس، best-value، ترسية مثلى، بوابة مورّد | `db/rfq.sql` + `db/rfq-portal.sql` + `rfq.html` |
| استيراد عرض بالذكاء | صورة عرض المورد → Gemini يملأ العمود | يحتاج الوسيط `GEMINI_API_KEY` |
| كشف الأسعار الشاذة | تنبيه عند تجاوز المتوسط التاريخي + إشعار المدراء | `proc_settings.anomaly_pct` |
| بطاقات أداء الموردين | تنافسية + استجابة + نشاط | محسوبة من البيانات |
| تذكيرات وتقارير مجدولة | انتهاء الوثائق، مواعيد RFQ، ملخّص أسبوعي | `db/cron.sql` (يلزم pg_cron) |
| التحقق الحكومي | وثق/Wathq للسجل التجاري | `functions/api/verify.js` (يلزم `WATHQ_API_KEY`) |

> كل الإشعارات داخل النظام (جرس) — بلا مورّد خارجي. الوصول للوحدات عبر زر الجرس.

## 🚀 قائمة تشغيل الوحدات (SQL — بالترتيب)
1. سكربت الإنشاء الأساسي (من واجهة الإعداد في `index.html`).
2. `db/hardened-rls.sql` (المرحلة 0 الآن، والمرحلة 1 بعد Supabase Auth).
3. `db/auth-migration.md` (خطوات Supabase Auth).
4. `db/workflows.sql` · `db/rfq.sql` · `db/rfq-portal.sql` · `db/secure-resume.sql`.
5. `db/cron.sql` (بعد تفعيل pg_cron).
